# ERP-Nexus Database Documentation

This folder contains the PostgreSQL database configuration, extensions, seed data, and performance indexes for **ERP-Nexus**. 

This schema is designed to work in synergy with the **Prisma ORM** models managed by the Express backend.

## 🗄️ Database Tables Reference

### 1. Authentication & User Profile Tables

#### `users`
* `id` (UUID, Primary Key): Autogenerated.
* `login_id` (VARCHAR(50), Unique, Not Null): User-chosen ID. Immutable.
* `email` (VARCHAR(254), Unique, Not Null): User's primary email. Admin-editable only.
* `password` (VARCHAR(128), Not Null): Hashed user password (bcrypt).
* `role` (Role Enum): Role authorization flag (`admin`, `sales`, `purchase`, `manufacturing`, `inventory`, `owner`).
* `is_active` (BOOLEAN, Default TRUE): Active status state.

#### `user_profiles`
* `profile_id` (UUID, Primary Key): Autogenerated.
* `user_id` (UUID, Unique, Foreign Key): Links to `users(id)` on cascade delete.
* `full_name`, `position`, `email_display`: Admin-editable only.
* `address`, `mobile_no`, `profile_photo`: Editable by user and admin.

#### `refresh_tokens`
* `token_id` (UUID, Primary Key): Autogenerated.
* `user_id` (UUID, Foreign Key): Links to `users(id)` on cascade delete.
* `token_hash` (TEXT, Not Null): Hashed token.
* `expires_at` (TIMESTAMPTZ, Not Null).
* `is_revoked` (BOOLEAN, Default FALSE).

### 2. Purchase Module Table Extensions

#### `products` (Universal Catalog)
Tracks inventory and procurement details for all raw materials, work-in-progress, and finished goods.
* `id` (UUID, Primary Key).
* `name` (VARCHAR(200), Not Null).
* `sales_price` / `cost_price` (DECIMAL(12,2), Not Null).
* `on_hand_qty` (DECIMAL(12,3), Default 0.000): Current physical inventory.
* `reserved_qty` (DECIMAL(12,3), Default 0.000): Stock reserved for confirmed sales orders.
* `reorder_level` (DECIMAL(12,3), Default 0.000): Stock threshold to trigger replenishment suggestions.
* `procurement_type` (ProcureType Enum): `MTS` (buy from vendor) or `MTO` (manufacture internally).
* `procure_on_demand` (BOOLEAN, Default FALSE).
* `vendor_id` (UUID, Foreign Key): Reference to the default supplier.

#### `purchase_orders` & `purchase_order_lines`
* `purchase_orders`: Header tracking vendor ID, status (`draft`, `confirmed`, `received`), and owner approvals.
* `purchase_order_lines`: Stores quantities ordered and received per line item.

#### `goods_receipts` & `goods_receipt_lines`
* `goods_receipts`: Logs receipt documents (`receipt_number`, expected `po_id`, and matching receiving user).
* `goods_receipt_lines`: Quantities actually received per product line.

#### `vendor_bills`
* Stores supplier invoices (`bill_number`, subtotal, tax, total_amount).
* Tracks payment execution (`status` enum: `pending_payment`, `approved_for_payment`, `paid`, `void`), attachment scans, and admin confirmation signatures.

#### `procurement_suggestions`
* Logs alerts generated when stock level (`on_hand_qty`) drops below `reorder_level` for MTS products.

## 🛡️ Trigger Automations & Business Logic

### 1. Immutability & Safety Triggers (Database Layer)
To protect structural records, database-level triggers in `002_triggers.sql` verify session credentials using the variable `erp.is_admin`:
* If `erp.is_admin` is not set to `'true'`, modifications to user roles, registrations, and core profile metadata are blocked.
* Updates to `users.login_id` are strictly prohibited for all connections.

### 2. Operational Business Logic (Application Layer)
Inventory mutations, stock ledger postings, and order status updates are managed by the **Node.js Express application** (`utils/stockMutations.js` and services) using Prisma transactions to ensure consistent business validations:
* **Goods Receipt Processing**: Upon receiving goods, the Express controller increments `products.on_hand_qty`, increments `purchase_order_lines.received_qty`, logs a movement in `stock_ledger` under type `purchase_in`, and promotes the PO status to `received` once complete.
* **Low Stock Alerts**: During inventory changes, the backend checks if `on_hand_qty < reorder_level` and creates a suggestion entry in the `procurement_suggestions` table if one is not already pending.
